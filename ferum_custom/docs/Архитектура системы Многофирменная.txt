Архитектура системы
    • Многофирменная архитектура и изоляция данных: Приложение поддерживает несколько компаний (юридических лиц) в единой ERPNext-среде. Для изоляции данных у каждого документа указывается Компания, и настроены User Permissions и условия запросов (Permission Query Conditions, PQC), чтобы пользователи видели только данные своей компании и своих клиентов[1][2]. Например, клиент с ролью Client видит только заявки, связанные со своим контрагентом (Customer), а инженер – только заявки, где он назначен исполнителем. Администратор (System Manager) имеет полный доступ ко всем данным[3].
    • Ролевая модель: Выделены основные роли: Administrator (администратор системы), Project Manager (руководитель проекта), Office Manager (офис-менеджер), Service Engineer (инженер) и Client (клиент). Каждая роль имеет чёткий набор прав на документы (см. раздел ниже) и доступ к функциям системы. Данные клиентов изолируются – внешние пользователи с ролью Client работают через портал/бота и не имеют доступа к внутреннему интерфейсу Desk[4]. Внутренние пользователи (менеджеры, инженеры) видят только проекты и заявки, к которым они относятся (по полям project_manager, assigned_to, customer и пр., с фильтрацией PQC)[1].
    • Общая архитектура: Система реализована на базе ERPNext (Frappe Framework) и кастом-приложения Ferum Custom. Основные данные и бизнес-логика хранятся в ERPNext (MariaDB/PostgreSQL) с использованием DocType и серверных скриптов Frappe[5]. В дополнение, существует отдельный бэкенд-сервис (на FastAPI/Python) для расширенных API и интеграций (Telegram-бот, Google API, JWT-аутентификация)[5][6]. Этот сервис использует общую базу данных или API Frappe для выполнения операций, обеспечивая дополнительный слой изоляции и возможность развертывания в контейнерах (Docker). Для веб-интерфейса используется стандартный Desk ERPNext для внутренних пользователей, а также React-фронтенд (и веб-портал ERPNext) для упрощённого интерфейса клиентов[7][6].
    • Изоляция окружений: Конфиденциальные параметры интеграций (учётные данные Google API, токен Telegram-бота, DSN Sentry и секрет JWT) хранятся отдельно для каждого окружения – в site_config.json или в едином DocType настроек (Ferum Custom Settings). Для разработки, тестирования и продакшена используются разные учетные записи и идентификаторы, чтобы гарантировать разделение данных. Например, в site_config.json на каждом сайте могут быть определены google_drive_root_folder_id, google_sheet_name, telegram_bot_token и др., либо эти параметры заполнены администратором через Ferum Custom Settings (Single DocType) в каждом окружении[8][9].
DocTypes: структура данных и ключевые поля
(Ниже перечислены основные DocType модуля ferum_custom с их назначением, полями и связями. Названия файлов указаны относительно папки приложения, например ferum_custom/ferum_custom/doctype/service_request/.)
Структура данных Ferum Custom: связи между основными DocType (ER-диаграмма). Например, один Service Project связан со многими Service Request и Invoice, Service Object могут повторно использоваться в проектах, а Service Request имеет связанный Service Report. Child-таблицы (внизу) представляют вложения и элементы.
    • Customer – стандартный DocType ERPNext Customer, представляющий клиента (компания или физлицо). Используется для привязки проектов и объектов. У клиента могут быть несколько проектов (Service Project) и объектов (Service Object)[10]. Ключевые поля: имя, тип, контактные данные (настраивается в ERPNext).
    • Service Project – кастомный DocType для сервисного проекта (договор на обслуживание)[11]. Файл: service_project/service_project.py и service_project.json. В полях хранится:
    • project_name – название проекта/договора.
    • customer – ссылка на Customer (клиент, заключивший договор)[12].
    • project_manager – ссылка на пользователя-менеджера проекта (роль Project Manager)[13]. Этот пользователь назначен ответственным за проект и обладает расширенными правами на связанные документы.
    • start_date, end_date – даты начала и окончания договора. Валидация не позволяет end_date быть раньше start_date[14].
    • status – статус проекта: Planned (планируется), Active (активен), Completed (завершён), Cancelled (отменён)[15]. Предусмотрен Workflow (файл Service Project Workflow в фикстурах) с переходами: активация, завершение, отмена, переоткрытие проекта. Например, из Planned менеджер может выполнить действие “Activate Project” для перевода в Active[16].
    • total_amount – сумма контракта (общая стоимость услуг по договору).
    • income_amount, expense_amount, profit_amount – вычисляемые поля финансов: суммарный доход, расходы и прибыль по проекту. Обновляются программно при изменении связанных счетов (Invoice): функция update_project_financials(project) пересчитывает эти поля суммируя оплаченные счета клиента и субподрядчиков[17][18].
    • objects – child-таблица (Project Object Item) со списком Service Object, охваченных этим проектом[19]. Каждый элемент таблицы имеет поле service_object (ссылка на Service Object)[20]. При сохранении проекта выполняется проверка validate_unique_objects: объект не должен дублироваться в списке и не может состоять одновременно в другом активном проекте[21][22].
    • attachments – таблица Document Attachment Item для файлов, прикреплённых к проекту (например, скан договора). Эти вложения также сохраняются на Google Drive (см. интеграции).
    • Права: Создавать проект могут Project Manager и админ[23]. Право на чтение имеют также Office Manager (для привязки заявок) и при необходимости – Department Head или Director (в матрице прав). Менеджер видит и редактирует только свои проекты (в коде get_permission_query_conditions возвращает проекты, где project_manager = текущий пользователь[24]).
    • Service Object – объект обслуживания, конкретное оборудование или локация у клиента[25]. Например, пожарная панель, система дымоудаления, адрес здания. Файл: service_object/service_object.py. Поля:
    • object_name – уникальное название объекта (например, Пожарная панель, здание А). Уникальность проверяется и кодом (метод validate не допускает дублей имен)[26].
    • customer – ссылка на клиента-владельца объекта. Обычно объект принадлежит одному Customer.
    • project – опциональная ссылка на Service Project, если объект сейчас на обслуживании по конкретному проекту/договору. Валидация обеспечивает, что объект не состоит одновременно в двух активных проектах (в ServiceProject.validate_unique_objects уже проверяется, и при удалении объекта – метод on_trash проверяет отсутствие активных проектов или открытых заявок)[27][28].
    • location/address, object_type – доп. информация о местонахождении и типе оборудования.
    • Права: Объекты могут создавать/редактировать менеджеры проекта (для внесения оборудования в договор) или админ. Клиентам доступна только просмотр своих объектов через портал. В PQC для Service Object прописано: проектные менеджеры видят объекты, если те включены в их проекты, а владельцы (создатели записи, например офис-менеджер, регистрирующий объект по звонку клиента) – видят свои записи[29].
    • Service Request – заявка на сервисное обслуживание (тикет). Ключевой DocType для регистрации обращений и отслеживания их выполнения[30]. Файл: service_request/service_request.py. Основные поля:
    • title – краткое название проблемы или работы.
    • description – подробное описание заявки.
    • type – тип заявки, например Routine Maintenance (плановое обслуживание) или Emergency (аварийный вызов).
    • priority – приоритет (High, Medium, Low), влияет на SLA.
    • customer – ссылка на Customer (определяется автоматически, когда указан Service Object)[31].
    • service_object – ссылка на Service Object, к которому относится заявка (может быть пустым, если заявка не на конкретное оборудование, тогда указывается вручную адрес/описание места). При выборе объекта система через set_customer_and_project() автозаполняет поля customer и project по данным объекта[31].
    • project – ссылка на Service Project (контракт/проект, под который подпадает заявка)[32]. Если объект привязан к проекту, заявка автоматически прикрепляется к этому проекту; иначе проект может быть выбран вручную (например, вне контракта).
    • assigned_to – назначенный исполнитель (пользователь с ролью Service Engineer). Этот пользователь отвечает за выполнение заявки. Назначение может быть сделано менеджером или автоматически (см. “Назначение ответственного” ниже).
    • status – статус заявки (Workflow): Open (открыта), In Progress (в работе), Completed (работы выполнены), Closed (закрыта), Cancelled (отменена). Статусы настраиваются через Workflow Service Request Workflow (фикстура). Начальный статус при создании – Open (устанавливается в before_insert)[33]. Переходы контролируются как Workflow-движком, так и дополнительной валидацией (метод validate_workflow_transitions). Особые правила: нельзя перевести заявку Open -> In Progress без назначенного инженера[34]; нельзя сменить In Progress -> Completed без привязанного отчёта (Service Report)[34]; закрыть заявку (Completed -> Closed) может только менеджер или админ[35]. Эти ограничения прописаны явно в коде[34]. При попытке нарушить правило выбрасывается ошибка (frappe.throw).
    • linked_report – ссылка на связанный Service Report (акт/отчёт о выполненных работах). Заполняется автоматически при сдаче отчёта: метод ServiceReport.on_submit устанавливает в связанной заявке этот поля и статус “Completed”[36]. Таким образом, при сабмите Service Report заявка автоматически помечается как выполненная.
    • reported_datetime – время создания/поступления заявки (обычно = creation или задаётся явно, если заявка поступила ранее).
    • actual_start_datetime, actual_end_datetime – фактическое время начала работ и завершения работ. Логика: при переводе статуса в In Progress система, если поле пустое, проставляет текущее время actual_start[37]; при переходе в Completed – проставляет actual_end текущим временем[38].
    • duration_hours – рассчитанное общее время выполнения (в часах). При наличии reported_datetime и actual_end_datetime вычисляется разница (метод update_timestamps)[39].
    • sla_deadline – крайний срок по SLA. При создании или обновлении приоритета рассчитывается calculate_sla_deadline() на основе типа и приоритета заявки[40]. Например, для Emergency с высоким приоритетом SLA = 4 часа от создания, для средних – 8 часов, для плановых – 1-3 дня[41].
    • attachments – child-таблица Request Photo Attachment Item (DocType для привязки файлов CustomAttachment). Она содержит список фото и документов, прикрепленных к заявке: фотофиксация, сканы и т.п.[30][42]. Каждая запись хранит ссылку на CustomAttachment.
    • События (скрипты):
        ◦ after_insert: после создания заявки система отправляет уведомление проектному менеджеру (notify_project_manager)[43]. Этот метод получает email менеджера проекта (get_project_manager_email) и шлёт письмо: “New Service Request {name} was created…”[44]. Ошибки логируются в случае недоставки[45].
        ◦ validate: выполняет set_customer_and_project (автозаполнение полей по объекту), validate_workflow_transitions (правила статусов, см. выше) и calculate_sla_deadline (SLA)[46].
        ◦ on_update: при каждом сохранении заявки проверяется нарушение SLA (check_sla_breach). Если заявка не завершена, а текущая дата превысила sla_deadline, система регистрирует нарушение: выводит сообщение и логирует ошибку SLA[47], а также ставит задачу (frappe.enqueue) на отправку уведомлений об эскалации (send_sla_breach_notifications)[48]. В том же on_update фиксируются временные метки (start/end, duration) через update_timestamps[37] и добавляется комментарий в таймлайн при смене статуса (для аудита)[49].
    • Права доступа:
        ◦ Project Manager – полный доступ (C/R/W) к заявкам своих проектов. Может создавать заявки (например, плановые работы) и назначать инженера, менять статус (кроме закрытия, если политика ограничивает).
        ◦ Service Engineer – видит и обновляет только заявки, на которых он назначен (assigned_to). Через PQC: ... where assigned_to = user[50]. Инженер может менять статус с Open на In Progress (при назначении на себя) и на Completed после выполнения, а также создавать черновик Service Report. Однако закрыть заявку (перевести Completed -> Closed) инженер не может – это делает PM или админ[51][52].
        ◦ Office Manager – может создавать заявки (например, по звонку клиента) от лица клиента, просматривать все заявки (для координации работ) и редактировать начальные стадии. Как правило, Office Manager не закрывает заявки и не помечает их выполненными, но может помогать в заполнении отчётов.
        ◦ Client – может создавать заявки через портал или бота, и видеть только свои заявки. В коде разрешено чтение заявок, где owner = текущий пользователь[53] (в случае, если клиент сам создал заявку через веб). Также на уровне ERPNext можно настроить User Permission привязку клиента к его Customer, чтобы он видел заявки только своего Customer[54][55]. В портале реализовано получение только своих заявок (функция list_service_requests фильтрует по owner для веб-пользователей)[56].
        ◦ Администратор – полный доступ ко всем заявкам (роль System Manager – без ограничений PQC)[57].
    • Service Report – отчёт о выполненных работах (акт). Каждая заявка в итоге должна иметь Service Report, описывающий что сделано[58]. Файл: service_report/service_report.py. Основные поля:
    • service_request – ссылка на связанную Service Request (обратная связь: один отчёт на одну заявку). Валидация гарантирует, что на одну заявку не будет два отчёта (проверяется уникальность связки или через Workflow).
    • report_date – дата составления отчёта. Если не указана, подставляется текущая дата при создании (before_insert)[59].
    • status – статус отчёта (Workflow Service Report Workflow): Draft, Submitted, Approved, Archived, Cancelled. Отчёт создаётся черновиком (Draft), затем после заполнения отправляется (Submit) на утверждение. Например, инженер создаёт черновик, PM или руководитель утверждает, после чего отчёт архивируется. Переходы ограничены: Draft -> Submitted, Submitted -> Approved, Approved -> Archived. Отмена (Cancelled) возможна только из состояния Submitted (после сабмита) и только для менеджера[60]. Эти правила прописаны в validate_workflow_transitions[61][62].
    • work_items – child-таблица Service Report Work Item – перечень выполненных работ и затрат[63]. Каждый Work Item содержит: описание работы, hours/qty – потраченное время или количество, rate – ставка (например, стоимость часа или ед. работы), total – рассчитанная стоимость по этому пункту. Поле total вычисляется автоматически: метод calculate_total_amount умножает hours на rate для каждого Work Item и суммирует в total_amount отчёта[64]. Примечание: в ТЗ требуется рассчитать total_payable – это может означать добавление отдельного поля для учёта выплачиваемой суммы (например, если ставка инженера и стоимость для клиента отличаются). В текущем коде total_amount представляет общую стоимость работ по акту для клиента, но может быть расширено: например, добавить поля internal_rate и total_payable для вычисления суммы оплаты инженеру/субподрядчику. Такой расчёт можно выполнять в этом же методе (hours * internal_rate).
    • documents – child-таблица Service Report Document Item – вложения к отчёту (сканы актов, фото отчётности)[65]. Каждый элемент содержит ссылку custom_attachment на документ (CustomAttachment). Валидация validate_attachments требует, чтобы для каждого указанного в таблице документа был прикреплён файл (непустое поле custom_attachment)[66] – иначе бросается ошибка, чтобы акт не был отправлен без необходимых приложений.
    • total_amount – общий итог по акту (сумма по всем Work Item, рассчитывается в calculate_total_amount)[64]. Может использоваться для сверки с суммой счёта клиенту.
    • События:
        ◦ validate: при сохранении черновика проверяет заполненность (через вызовы методов выше), корректность перехода статусов (validate_workflow_transitions) и наличие хотя бы одного Work Item (validate_work_items)[67][68]. Требование: нельзя подать отчёт без указания работ (предотвращает пустые акты).
        ◦ on_submit: при проведении (submit) отчёта выполняет update_service_request_on_submit – обновляет связанную заявку: записывает в неё linked_report = имя отчёта и переводит статус заявки в “Completed”[36]. Таким образом, акт автоматически помечает заявку выполненной (нельзя закрыть заявку без акта). Далее, on_submit ставит задачу загрузки PDF в Drive: вызывает enqueue_drive_upload()[69]. Этот метод через frappe.enqueue ставит в очередь фоновую задачу _upload_report_pdf[70].
        ◦ _upload_report_pdf(docname): генерирует PDF-файл отчёта и загружает его на Google Drive через API[71][72]. Путь на Google Drive строится как /{Customer}/{Project}/Reports/ServiceReport-{name}.pdf – т.е. используется структура папок: папка клиента, внутри – папка проекта, внутри – каталог Reports для актов[73]. PDF получается функцией frappe.get_print по DocType Service Report, затем байты передаются в integrations.drive.upload_bytes()[72]. Загрузка выполняется асинхронно (очередь short). Если загрузка не удалась, ошибка логируется в frappe.log_error. Примечание: Требование «учёт Drive File ID и обновление PDF без дублирования» означает, что система должна хранить ID загруженного на Диск файла и при повторной загрузке того же отчёта (например, если отчёт отозван и заново утверждён) – обновлять существующий файл, а не создавать дубликат. В текущей реализации upload_bytes возвращает ID файла[74], и целесообразно сохранить его в CustomAttachment, привязанном к ServiceReportDocumentItem. План доработки: при повторном сабмите акта проверять, есть ли уже файл с таким ID, и заменять его содержимое (либо удалять старый перед загрузкой нового).
        ◦ on_update: логирует изменение статуса акта – добавляет комментарий, если статус изменился[75].
    • Права: Инженер и PM могут создавать черновик Service Report для заявки, к которой имеют доступ. В матрице указано: Engineer – C/R (создавать и читать) свои акты, PM – C/R/W акты по своим проектам[76][77]. После сабмита акта (docstatus=1, статус Submitted/Approved) править его могут только админ или менеджер (например, если нужно отменить). Permission Query Condition для Service Report допускает просмотр: PM – если он менеджер проекта, к которому относится заявка этого акта, инженер – если он назначен в заявке[78]. Функция has_permission даёт доступ PM и инженеру к конкретному документу при тех же условиях[79][80]. Клиентам по умолчанию Service Report не показывается (они получают итоговый PDF по почте или через портал по ссылке, либо доступ к файлу); однако опционно можно разрешить клиенту читать финальные отчёты, ограничив их проекты – например, через условие “customer = client’s customer”.
    • CustomAttachment – вспомогательный DocType для хранения метаданных файлов (кастомный аналог стандартного File). Файл: custom_attachment/custom_attachment.json (структура описана в дизайне). В системе все прикрепляемые файлы (фото, сканы актов, счета) сохраняются не просто как File, а как CustomAttachment для расширенного контроля. Поля: file_url (ссылка на файл или ID Google Drive), file_name, content_type, размер и т.п. Важно: CustomAttachment используется только как хранилище ссылок, а привязка к основным документам идёт через child-таблицы (RequestPhotoAttachmentItem, ServiceReportDocumentItem, InvoiceAttachmentItem и т.д.), что позволяет одному файлу при необходимости быть привязанным в разных контекстах[81]. Права: те же, что и на родительский документ; удалять файл может только тот, кто может редактировать родителя (в коде это контролируется, например, при удалении attachments у заявки или через общие правила).
    • Invoice – кастомный DocType для учёта счетов, как выставляемых клиенту, так и полученных от субподрядчиков[82]. Он фактически объединяет функциональность Sales Invoice (счёт клиенту) и Purchase Invoice/Bill (счёт от поставщика) в одном документе для удобства внутреннего мониторинга. Файл: invoice/invoice.py. Основные поля:
    • project – ссылка на Service Project, к которому относится счёт (обычно каждый счёт привязывается к конкретному проекту/договору)[83]. Может быть пустым, если счёт вне контекста проекта (редко, например, общие расходы).
    • counterparty_type – тип контрагента: Customer (если это счёт клиенту на оплату наших услуг) или Subcontractor (если это входящий счёт от субподрядчика, подлежащий оплате)[84].
    • counterparty_name – наименование контрагента. Для типа Customer это, как правило, ссылка на Customer (имя компании-клиента), для Subcontractor – название подрядчика или ФИО специалиста. В JSON это может быть Link на Customer или Supplier, либо просто Data поле.
    • amount – сумма счёта (в валюте, по умолчанию валюта компании, возможно поле currency).
    • invoice_date – дата счёта или период (например, месяц, за который выставлен счёт). Иногда используется вместо поля month.
    • status – статус счёта: например, New/Draft (черновик), Sent (отправлен клиенту или получен от подрядчика и ожидает оплаты), Paid (оплачен), Cancelled (отменён). Workflow Invoice Workflow определяет возможные переходы: например, менеджер или офис-менеджер помечают счёт как “Sent” после проверки, а бухгалтер/админ – как “Paid” после фактической оплаты. В коде реализована массовая операция bulk_mark_sent для установки статуса Sent на несколько счетов сразу (доступна ролям System Manager, Project Manager, Office Manager, Chief Accountant)[85][86]. Установка статуса Paid предполагает, что документ проведён (docstatus=1) и имеется право у текущего пользователя (обычно Chief Accountant или System Manager).
    • sales_invoice – опциональная ссылка на стандартный документ Sales Invoice ERPNext. Если включена опция enable_auto_create_sales_invoice в настройках, то при переводе нашего Invoice (counterparty_type = Customer) в статус “Sent” или “Paid” система автоматически создаёт в ERPNext обычный Sales Invoice и привязывает его, чтобы отразить в бухгалтерском учёте продажу[87]. Это делает функция create_sales_invoice(invoice_name) (непоказана выше), вызываемая в on_invoice_update[88].
    • attachments – child-таблица (например, Invoice Attachment Item), хранящая ссылки на CustomAttachment (скан счёта, актов, подтверждений оплаты и т.д.). Загрузка этих файлов также идёт в Google Drive (в папки, например, /Project/Invoices).
    • Скрипты и интеграции:
        ◦ after_insert: когда создан новый Invoice, запускаются два важных процесса: синхронизация с Google Sheets и уведомления. В функции after_insert вызывается notify_on_subcontractor_invoice() и ставится в очередь задача sync_to_google_sheets[89].
        ◦ notify_on_subcontractor_invoice: если counterparty_type = Subcontractor, система формирует сообщение: “New subcontractor invoice {name} for {counterparty_name}. Amount: {amount}”[90]. Это сообщение отправляется: а) в Telegram (в дефолтный чат, например группа админов) через tg_send()[91][92]; б) собирается список email-адресатов – роли, указанные в настройках invoice_notification_roles (по умолчанию Chief Accountant и System Manager) – и рассылается email всем им[93]. Цель – оперативно уведомить ответственных, что получен новый счёт от подрядчика, требующий проверки/оплаты. Если никого не найдено, сообщение логируется как предупреждение[94].
        ◦ sync_to_google_sheets(docname): фоновая задача для односторонней синхронизации данных счета в Google Sheets[95]. Используется библиотека gspread с сервисным аккаунтом. Функция get_google_sheet() открывает Google Sheet по имени из настроек (например, “Ferum Invoices Tracker”) и берет первый лист[96][97]. В sync_to_google_sheets происходит: получить документ Invoice, найти в таблице строку с таким именем (sheet.find), сформировать массив данных: [ID счета, Проект, Имя контрагента, Тип, Сумма, Статус, Дата, timestamp синхронизации][98]. Если запись уже была, обновить строку (sheet.update по координате)[99], если нет – добавить новую строку (sheet.append_row)[100]. В обоих случаях пользователю показывается MsgPrint об успехе синхронизации[99]. При ошибках – логируются через frappe.log_error[101].
        ◦ Подсветка и проверка роли загрузившего: Само форматирование (например, выделение цветом строк, где счет внёс не PM проекта) реализовано на стороне Google Sheets – либо через формулы, либо через скрипт. Наш sync передаёт столбец Entered By или counterparty_type для условного форматирования. В ТЗ упомянуто “с подсветкой строк, проверкой роли загрузившего” – это соответствует описанию: лист имеет правило подсветить, если строку внёс не Project Manager данного проекта (сравнивается имя PM проекта и пользователь-создатель счета)[102][103]. Наш код предоставляет для этого нужные данные (в поле counterparty_type явно видно, клиентский это счёт или субподряд, а через frappe.session.user можно определить, является ли он PM проекта). Таким образом, при заполнении строки можно добавить имя создавшего и имя PM, а форматирование (например, красная заливка) в Google Sheets выделит, если они не совпадают[102].
        ◦ on_update (и on_cancel): в hooks настроено, что при любом обновлении или отмене Invoice вызывается функция on_invoice_update[104]. Она выполняет: пересчёт финансов проекта (update_project_financials) если указан проект[105]; при переводе счета в Paid (docstatus=1 и status="Paid") или при отмене (docstatus=2) – отправляет обновление в Google Sheets очередью (чтобы зафиксировать оплату или убрать счет)[106]. Также внутри on_invoice_update реализовано автоматическое создание Sales Invoice (см. выше)[87].
    • Права:
        ◦ Создавать Invoice могут Project Manager (для счетов по своим проектам, как клиентских, так и загружать подрядчиков), Office Manager (помощь в заведении счетов – им разрешено создавать счета по любым проектам, т.к. они поддерживают отделы) и Admin/Chief Accountant[107].
        ◦ Править черновик счета (например, добавлять скан, менять сумму) может PM, офис-менеджер или админ. Однако изменение статуса и проведение – ограничено: только Chief Accountant или Administrator помечают счет оплаченным (Paid)[108]. Можно реализовать это через Workflow: действие “Mark Paid” доступно только ролям Chief Accountant/System Manager, или через поле status с role permission (например, сделать его read-only для всех кроме бухгалтера).
        ◦ Чтение: Project Manager видит счета своих проектов, Office Manager – всех (для координации документооборота), Chief Accountant – всех, Client – не имеет доступа к этим внутренним счетам (клиент получает свой счёт по email отдельно). Исключение: при желании, можно добавлять клиенту отдельный DocType “Customer Invoice” с выпиской, но в данной системе клиенты не просматривают Invoice напрямую[109].
    • PayrollEntryCustom – (если включён HR-модуль) – кастомный DocType для расчёта зарплат. Он расширяет стандартный Payroll Entry. Поля: период, список сотрудников с начислениями, общие суммы, учёт авансов. В текущем контексте HR не является фокусом, но упоминается, что PayrollEntryCustom содержит total_payable по каждому сотруднику (зарплата к выплате, с учётом авансов)[110]. В рамках модуля Ferum Custom он используется для учёта трудозатрат: например, часы инженеров из ServiceReport могут агрегироваться для зарплаты. Однако конкретные связи (например, связка ServiceReportWorkItem -> Employee) не детализированы. Права: доступны только Chief Accountant/HR и администратору (конфиденциальные данные)[111].
(Примечание: Помимо вышеописанных, в модуле есть второстепенные DocTypes, например, Service Maintenance Schedule (график плановых заявок) – ежедневный планировщик создаёт по нему заявки автоматически (см. hooks generate_service_requests_from_schedule в scheduler_events) и т.п. Они работают по аналогии с описанными компонентами, но в ТЗ основное внимание уделяется приведённым сущностям.)
Скрипты бизнес-логики (события DocType и функции)
Каждый DocType снабжён серверными скриптами (методами) в файлах Python, обеспечивающими требования бизнес-логики. Ниже перечислены ключевые функции/методы и триггеры:
    • Назначение ответственного инженера (авто-назначение): При создании новой Service Request из портала или офис-менеджером, система может предложить или назначить инженера на основе объекта или проекта. Для этого планируется реализовать функцию get_engineers_for_service_object(service_object) (например, в ferum_custom.utils), которая возвращает список подходящих инженеров. Логика подбора: ищет всех пользователей с ролью Service Engineer, которые связаны с данным объектом или проектом. Например, можно хранить в Service Object поле “Preferred Engineer” или анализировать последние заявки по этому объекту и выбрать инженера, который ранее обслуживал этот объект. Пока это делается вручную менеджером, но требование предполагает автоматизацию. Возможная реализация: при выборе Service Object в форме заявки (через клиентский скрипт) вызывать frappe.call метода, который вернёт рекомендованного инженера, и заполнить поле assigned_to. Если объект входит в проект, можно привязать инженеров, назначенных на проект (в будущем – добавить к Service Project child-таблицу “Project Team” с инженерами). В текущей версии подобного поля нет, поэтому функция может просто возвращать всех активных инженеров или инженера из прошлой заявки на этот объект. Затем менеджер подтверждает назначение. В коде validate_workflow_transitions уже есть проверка: статус In Progress нельзя установить без assigned_to[34], это гарантирует, что каждая заявка имеет ответственного.
    • SLA и эскалации: За каждым Service Request ведётся отсчёт времени по SLA. При создании заявки вычисляется дедлайн sla_deadline согласно приоритету/типу[40]. Фоновый процесс (hook hourly – check_all_slas) каждый час проверяет все открытые заявки и вызывает check_sla_breach()[112][113]. Если текущая дата превышает sla_deadline и заявка всё ещё Open/In Progress, система фиксирует нарушение SLA:
    • Локально – отображает предупреждение (msgprint) ответственному на Desk и логирует инцидент в журнал ошибок (SLA Breach Alert)[114].
    • Инициирует отправку уведомлений: через frappe.enqueue вызывает send_sla_breach_notifications(service_request, message)[48]. Эта функция собирает получателей: менеджер проекта заявки (если есть) и всех Office Manager (через get_users_by_roles(["Office Manager"]))[115], а затем отправляет им email с темой “SLA breached for Service Request X” и текстом сообщения[116]. Дополнительно пытается отправить уведомление в Telegram – через tg_send(message) отправляет в дефолтный чат (например, группа менеджеров) уведомление о просрочке[117]. В случае ошибок при отправке Telegram – подавляет исключение, чтобы не прерывать процесс[118]. В итоге ответственные лица немедленно узнают о просрочке (эскалация).
    • Данные о нарушениях SLA могут логироваться структурированно (например, накапливать в отдельной таблице “SLA Log” или в Sentry) для последующего анализа (сколько нарушений, по каким причинам). Также вычисляется duration_hours выполнения каждой заявки – эти данные используются для KPI (среднее время закрытия, процент вовремя закрытых заявок и т.д.)[119].
    • Валидации Workflow и статусов: Для ключевых DocType реализованы методы проверки корректности переходов состояний (как описано):
    • Service Request: validate_workflow_transitions не даёт нарушить последовательность “Open -> In Progress -> Completed -> Closed” и проверяет наличие ответственного и отчёта[34].
    • Service Report: аналогично, validate_workflow_transitions контролирует переходы статусов отчёта (Draft->Submitted->Approved->Archived, отмена)[61][62], и validate_work_items требует список работ[68].
    • Invoice: нет явного метода валидации, но при сохранении можно добавить: требовать заполненные проект, сумму, контрагента; блокировать смену статуса на “Paid” без docstatus=1 и соответствующих прав. Workflow Invoice (настроен через фикстуру) определяет допустимые действия (например, “Send” для пометки Sent, “Mark Paid” для Paid доступны ограниченным ролям[120]).
    • Авто-расчёты сумм:
    • ServiceReport: функция calculate_total_amount при каждом сохранении пересчитывает item.total и суммарно doc.total_amount[64]. Этот метод вызывается и на validate, гарантируя актуальность сумм. Требуемое расширение – расчет total_payable (например, добавить в Service Report Work Item поля pay_rate и total_payable для случаев, когда оплата инженеру отличается от тарифа клиенту). Такой расчёт можно включить в calculate_total_amount: параллельно суммировать hours * pay_rate в total_payable_amount отчёта. Эти данные могут затем использоваться для заполнения PayrollEntryCustom.
    • PayrollEntryCustom: При сохранении можно автоматически суммировать по каждому сотруднику: брутто-зарплата, авансы (если ввести поля advance_paid) и рассчитывать net_pay (к выплате). В описании упомянуто, что скрипт вычитает аванс из начислений[121].
    • DocEvents hooks:
    • В hooks.py настроены doc_events для Invoice: привязка on_update и on_cancel к функции on_invoice_update[104], чтобы перерасчитать финансы проекта и синхронизировать Google Sheet при каждом изменении статуса счета (например, пометка “Paid” должна сразу отразиться в таблице)[106].
    • Scheduler Events: Ежедневно (daily) запускается генерация заявок из расписания (service_maintenance_schedule.generate_service_requests_from_schedule) – т.е. по объектам, где запланированы регулярные работы, создаются Service Request автоматически[122]. Ежечасно (hourly) – проверка SLA (check_all_slas)[113]. Эти фоновые задачи обеспечивают проактивную работу системы.
    • Логирование действий: Многие методы добавляют comment (типа Info) в timeline документов при важных событиях: изменение статуса Service Request, Service Project, Service Report – см. вызовы add_comment("Status changed to ...")[49][123]. Это обеспечивает аудит действий: кто и когда поменял статус. Кроме того, функция frappe.get_doc().add_comment автоматически пишет пользователя. А ERPNext ведёт Version-лог изменений значимых полей, включён track_changes во всех DocType (см. JSON track_changes: 1)[124].
Роли и права доступа
(Сводка по ролям из матрицы прав и их возможности:)
    • Administrator (System Manager): Полный суперадмин-доступ ко всем функциям и документам системы[3]. Может создавать и настраивать пользователей, менять любые данные, выполнять закрытие заявок и проведение документов вне зависимости от ограничений. Только администратор может менять настройки системы (например, токены интеграций). Также админ получает особые уведомления (например, о новых счетах, ошибках), и должен подтверждать критические действия (финальное закрытие контрактов, пометка счетов оплаченными в отсутствие бухгалтера).
    • Project Manager: Ведет конкретные проекты обслуживания. Основные права: создавать и редактировать Service Project (только свои проекты), добавлять в них объекты; создавать Service Request внутри своих проектов, назначать инженеров; просматривать все заявки и отчёты по своим проектам, редактировать их статусы (Open->In Progress->Completed). PM может сам оформить Service Report (например, если инженер продиктовал результаты) либо отредактировать черновик, поданный инженером, и затем утвердить/подписать акт[125][126]. PM также может инициировать создание счетов (Invoice) по своему проекту – как клиентских (выставить счёт клиенту, подготовив данные для бухгалтера), так и загружать счета от подрядчиков на свой проект[127][107]. Однако PM не отмечает счета “оплаченными” сам – это делает бухгалтер/админ. По умолчанию PM не видит чужие проекты и связанные документы (PQC фильтрует по полю project_manager = user в Service Project/Request/Report)[128][78].
    • Office Manager: Административная роль, поддерживающая документооборот. Имеет права создавать заявки (когда клиенты звонят или присылают email – офис-менеджер регистрирует Service Request от их имени)[129]. Может помогать заполнять акты (Service Report) – например, инженер привёз бумажный акт, офис-менеджер вводит его в систему и прикрепляет сканы. Также Office Manager может заносить счета: часто бухгалтерия поручает офис-менеджеру внести поступившие счета от субподрядчиков или сформировать черновик счёта для клиента, который потом проверит PM[130][131]. Чтение: офис-менеджер видит практически все документы по всем проектам (кроме сугубо HR). Он нужен для координации – например, знать все открытые заявки, чтобы сообщать статус клиентам при звонке, следить за графиками ТО. Однако некоторые действия офис-менеджеру недоступны: он не закрывает заявки (только помечает Open), не утверждает контракты и не меняет финансовые статусы.
    • Service Engineer: Исполнитель работ на местах[132]. Доступ инженера ограничен только нужными ему данными: он видит свои назначенные заявки (и, возможно, все открытые заявки без назначенного, если реализован механизм “самоназначения” – сейчас не реализовано, но потенциально можно позволить инженерам брать свободные заявки). Инженер обновляет статус своих заявок (In Progress, Completed)[51], не заходя при этом в чужие заявки. Он также создает Service Report (черновик) по выполненной заявке: описывает работы, часы, добавляет фото “до/после”. После этого либо сам инженер сабмитит акт (если компания доверяет) либо сообщает менеджеру о готовности. Инженер может прикреплять файлы (фотофиксацию, документы) к заявке и акту – через веб или мобильный бот. Финансовые данные ему недоступны: он не видит счета, суммы контрактов, прибыль по проектам и т.д.[133]. Также инженер не имеет доступа к HR-записям (кроме своих, если реализована личная инфо).
    • Client (Customer пользователь): Внешний пользователь, обычно представитель клиента, обладающий минимальным доступом через портал или чат-бот[4]. Клиент может: создавать новые заявки на обслуживание (через веб-форму /portal/new-request или команду бота)[134], просматривать статус своих заявок (список и детали через /portal или бот-команды /my_requests, /request_status)[135]. Также клиент может скачать итоговые документы: например, после закрытия заявки видеть ссылку “Download Service Report” у себя в портале[136] – реализовано, что если linked_report заполнен, на портале выводится ссылка на PDF акта[136]. Клиент не видит внутренние поля (например, assigned_to, затраты, счета) – только информацию, необходимую ему (описание, статус). Он не имеет доступа ни к одному документу, где Customer не является его компанией. За это отвечает как PQC (условие Service Request.owner = user для портала) и/или User Permission, привязывающая данного пользователя только к своему Customer[137][138]. Таким образом, даже если клиент попробует получить доступ по прямой ссылке к чужому документу, система отфильтрует запрос.
    • Chief Accountant: (упоминается в матрице, хотя не был явно в списке, но можно включить как подвид администратора) – Главный бухгалтер/финансовый менеджер. Он обладает правами на Invoice и Payroll: создаёт и проводит PayrollEntryCustom, просматривает и помечает счета оплаченными[139][140]. В контексте заданных ролей его функции могут выполнять администратор, но если роль есть, она ограничена финансами. Бухгалтер не правит заявки и отчёты, но может их просматривать для финансового контроля.
(Примечание: Также в системе предусмотрены роли General Director (генеральный директор) – с правом утверждения крупных операций, и Department Head – руководители подразделений. Они фигурируют в фикстурах ролей[141]. Например, директор имеет право окончательного “Approval” проектов, контрактов и крупных платежей (его роль можно задать в Workflow состояний “Pending Approval” -> “Approved”). Эти роли имеют широкое чтение, но мало прямого редактирования – в рамках данного ТЗ они не основной фокус, однако система готова к их добавлению (в Workflow состояний фигурируют “Pending Approval” и др. для таких ролей)[142][143].*
    • Контроль доступа (PQC и User Permissions):
Все вышеперечисленные ограничения реализованы комбинацией настроек прав DocType и Permission Query Conditions в Python. Например, в service_request.py определён get_permission_query_conditions: для Project Manager добавляет условие EXISTS (Service Project с project_manager = текущий пользователь)[144], для Service Engineer – условие по assigned_to, для Client – по owner[50]. Аналогично для Service Report, Service Project, Service Object определены свои PQC функции[78][29]. Это означает, что даже на уровне списка в Desk пользователь не увидит чужих записей. Дополнительно, User Permissions можно использовать для привязки пользователя к разрешённым записям: напр., каждому клиентскому пользователю в ERPNext настраивается User Permission на свой Customer, каждому инженеру – на свою Employee или Company при мульти-компании. Также, на уровне Roles Permission Manager настроены галочки Create/Read/Write/Submit для каждой роли по каждому DocType (см. JSON permissions в DocType). Пример из JSON Service Project: PM и Office Manager имеют Create/Read/Write, инженер – только Read[145][146]. Такими настройками обеспечивается, что, например, инженер не сможет создать или удалить проект, а клиент не сможет редактировать заявку после создания и т.д.
    • Политики безопасности: Для особо важных пользователей (админ, бухгалтер) включена двухфакторная аутентификация при входе (через стандартный 2FA ERPNext с OTP)[147]. Это гарантирует, что даже при компрометации пароля злоумышленник не получит доступ к данным. Доступ клиентов ограничен веб-порталом – они не имеют учётных записей Desk User (в User записано user_type = Website User). В hooks присутствует скрипт User.user_type (public/js/user.js) – возможно, он автоматически переключает тип пользователя при назначении ролей (чтобы клиентские аккаунты оставались только портал-пользователями)[148].
Интеграции внешние системы
(Модуль Ferum Custom тесно интегрируется с сервисами Google, мессенджерами и внешними API. Все интеграции реализованы с учётом изоляции (раздельные ключи для тест/прод) и безопасности (хранение секретов вне кода). Ниже по каждому направлению:)
    • Google Drive (Хранение файлов): В системе практически все файлы (фото, сканы актов, счета) сохраняются в облачном хранилище Google Drive, а не на сервере ERPNext[149][150]. Это сделано для разгрузки базы и удобного доступа. Реализовано через сервисный аккаунт Google:
    • В Ferum Custom Settings на каждом сайте указывается google_service_account_json – путь до JSON с учётными данными сервисного аккаунта Google[151]. Этот JSON загружается администратором как файл (например, в ERPNext File) и привязывается в настройки, после чего код загружает его и строит credentials (Google OAuth2)[151][152].
    • Базовая функция upload_bytes(path_parts, filename, data, mime_type) (файл integrations/drive.py) выполняет загрузку массива байт на Google Drive[153][154]. Она использует Google Drive API (библиотека googleapiclient) с правами drive.file[155]. Алгоритм:
        a. Получает корневой ID папки из настроек (google_drive_root_folder_id) – это ID заранее созданной папки, например “FerumFiles”, под которой будут создаваться структуры[154].
        b. Проходит по списку path_parts – имен подпапок – и для каждой вызывает _ensure_folder(drive, name, parent_id). Эта вспомогательная функция ищет папку с данным именем под указанным родителем, если нет – создает[156][157]. Таким образом, динамически строится дерево папок: например, [Customer, Project, Reports].
        c. После обеспечения папки загружает файл: формирует MediaInMemoryUpload из переданных байт и вызывает drive.files().create(body={name: filename, parents: [parent_id]}, media_body=media, fields="id")[158]. В ответ получает file ID.
        d. Возвращает ID файла (или None при ошибке). Все ошибки ловятся и логируются через frappe.log_error[159][160].
    • Прикрепление файлов из ERPNext UI (например, добавление фото к заявке) можно делать через CustomAttachment. Вероятный сценарий: когда пользователь загружает файл, Frappe сначала сохраняет его временно, затем скрипт (например, File.after_insert hook) вызывает логику, аналогичную upload_bytes. В текущем коде явно не показан перехват File, но можно реализовать File Hook или переопределить attach_print для PDF. В случае Service Report – PDF генерируется явно, без сохранения локально, сразу отправляется на Drive.
    • Учет ID и обновление: планировалось, что CustomAttachment будет хранить поле drive_file_id. Например, при загрузке PDF акта функция _upload_report_pdf после upload_bytes должна создать запись CustomAttachment с этим ID и прикрепить её в Service Report Document Item. Тогда, при изменении (если акт перегенерирован), можно найти по ServiceReport связанный CustomAttachment и вызвать Drive API files().update(fileId=..., newMedia=...) для замены содержимого (либо удалить и заново создать, хотя Drive позволяет заменять контент файла). Это предстоит доработать согласно требованию “без дублирования файлов”.
    • Доступ и безопасность: Все файлы хранятся в приватной папке сервисного аккаунта – по умолчанию они не доступны публично. Когда пользователь в ERPNext открывает вложение, система может либо: а) по ID файла получить через API содержимое и отдать его пользователю (нужна функция-прокси, проверяющая права и скачивающая файл), либо б) сделать ссылку “для просмотра по ссылке”. Более безопасно – первый вариант. Пока, например, ссылка на PDF Service Report на портале реализована через генерацию PDF из ERPNext напрямую[136]. В будущем можно сделать выдачу именно Drive-версии. Удаление файла: если пользователь удаляет вложение в ERPNext, желательно вызывать Drive API для удаления соответствующего файла (можно реализовать в CustomAttachment.on_trash). Аналогично, обновление – обновлять, а не плодить версии.
    • Итог: интеграция с Drive гарантирует централизованное хранение файлов (Google инфраструктура), резервирование их на Google side. Администратор в site_config указывает google_drive_root_folder_id для продакшена и может другой для теста, чтобы не смешивать файлы.
    • Google Sheets (Синхронизация счетов): Предусмотрена односторонняя интеграция: данные о счетах из ERPNext экспортируются в Google Spreadsheet “Invoice Tracker”[161][162]. Это нужно для финансового обзора и свода. Ключевые моменты:
    • Настройки: google_sheet_name или ID можно задать в Ferum Custom Settings. Код get_google_sheet() использует gspread с credentials сервисного аккаунта (с scope spreadsheets)[163]. Он открывает гугл-док по имени (например, “Ferum Invoices Tracker”) и берет первый лист[164]. Важно: для изоляции окружений можно указать разные имена/ID таблицы в настройках dev/prod.
    • Данные: при создании или обновлении Invoice, функция sync_to_google_sheets обновляет таблицу: либо добавляет строку с новым счетом, либо обновляет существующую, найденную по ID счета[165][166]. Передаются поля: ID, проект, название контрагента, тип (Customer/Subcontractor), сумма, статус, дата, timestamp обновления. В Google Sheet настроены формулы и сводные таблицы: суммирование по проектам и по месяцам, подсветка аномалий (например, подсвечивать, если EnteredBy != ProjectManager)[167][102]. Это позволяет менеджменту быстро видеть, сколько выставлено клиенту и сколько нужно платить подрядчикам по каждому проекту, без захода в ERP.
    • Обратная синхронизация: текущая интеграция – в одну сторону (ERPNext -> Sheets)[168]. Обновление статуса оплаты может фиксироваться вручную в ERPNext и листе (как сейчас: пометил Paid в ERP, это ушло в лист). Можно рассматривать и обратный путь: например, отметить оплату в листе – чтобы ERPNext подтянул изменение. Пока от этого отказались для упрощения (истина хранится в ERPNext).
    • Ошибки и очередь: Вставка строк в Google API – не мгновенная операция, поэтому отправляется через enqueue (очередь short) при создании и важных обновлениях счетов[169][170]. Если Google API временно недоступен, функция sync_to_google_sheets ловит исключения и логирует их (в Error Log)[101], чтобы админ мог увидеть сбой. Можно также реализовать повтор попытки: например, если вставка не удалась, сохранить флаг и попробовать позже (но пока это не сделано, кроме как ручной повтор).
    • Безопасность: Учётные данные – тот же сервисный аккаунт (JSON) с правом на Drive и Sheets. Файл таблицы должен быть расшарен на этот сервисный аккаунт (по email). Доступ к таблице для людей: обычно в компании таблицу открывают на просмотр PM, офис-менеджеру, бухгалтерии – через обычный Google Share, или даже публикуют (но лучше ограничить).
    • Telegram Bot (Уведомления и команды): Интеграция с Telegram реализована для оперативных уведомлений и взаимодействия пользователей (инженеров, клиентов) через чат-бот. Основные моменты:
    • Уведомления (исходящие): При определенных событиях система шлет сообщения через Telegram Bot API. Используется библиотека requests – через HTTP-вызов на https://api.telegram.org/bot<TOKEN>/sendMessage[171]. Токен бота хранится в настройках telegram_bot_token, ID целевого чата для групповых уведомлений – в telegram_default_chat_id[9]. Функция send_message(text, chat_id=None) берет эти настройки, и если они заданы, отправляет сообщение (с несколькими ретраями на случай временного сбоя)[172][173]. В текущем коде используется, например:
        ◦ Новый счёт подрядчика: отправка уведомления в группу админов (chat_id по умолчанию)[90].
        ◦ Нарушение SLA: отправка alert-сообщения (возможно, тоже в группу диспетчеров)[117].
        ◦ Новая заявка: можно настроить, что при появлении Service Request бот пишет в командный чат инженеров: “Новая заявка №X от Клиента Y: ...”. Пока явно не показано, но в интеграционном документе это предусмотрено[174].
        ◦ Отчёт выполнен: уведомление менеджеру/админу, что инженер загрузил Service Report и заявка завершена[175].
        ◦ И т.д. Все такие уведомления отправляются сервером, вызывая telegram.send_message. Бот, привязанный к токену, пересылает их адресатам. В случае ошибки (неверный токен, недоступный интернет) – send_message логирует предупреждение и возвращает False[176][177]. Таким образом, сбой Telegram не ломает бизнес-логику (есть fallback на email в важных случаях, например, SLA и счета дублируются email-уведомлением)[115][93].
    • Команды (входящие): Отдельно реализован Telegram Bot Handler для обработки команд от пользователей. Судя по документации, предполагается использование библиотеки Aiogram с асинхронным прослушиванием сообщений[178]. Этот бот-сервис, вероятно, работает в контексте отдельного бэкенда (FastAPI). В ferum_custom репозитории можно не найти реализации команд, значит она находится вне (либо планируется). Тем не менее, проектная документация описывает такие команды:
        ◦ /new_request – клиент инициирует новую заявку через бота. Бот запрашивает детали (в диалоге) – объект, описание – и затем вызывает REST API POST /requests на нашем бэкенде, который создает Service Request (аналог ferum_custom.api.service.create_service_request)[179][180]. Клиент получает ответ “Заявка создана №...”.
        ◦ /my_requests – инженер запрашивает список своих открытых заявок. Бот вызывает GET /requests?assigned_to=me (или использует JWT токен инженера) и формирует список в сообщении[181].
        ◦ /request_status <ID> – получить статус и информацию по заявке №, бот тянет GET /requests/ID и отвечает с полями status, title, etc.
        ◦ /start <ID> – (или /set_status) – инженер отмечает начало работы. Бот вызывает PUT /requests/ID/status с телом {"status": "In Progress"}[182][183]. Сервер проверяет JWT (который принадлежит инженеру) и право – заявка должна быть назначена этому инженеру, иначе вернёт 403. Если всё ок, статус обновится, а бот ответит “Заявка №X в работе”.
        ◦ /done <ID> – аналогично, запрос на завершение. Бот пытается перевести заявку в Completed. Если в системе настроено требование акта, сервер вернет ошибку “Service report required”[184]. Тогда бот может инициировать сбор данных для акта: попросить инженера отправить текст отчёта и фото. Возможная реализация: бот принимает фото (/upload_photo ID – и просто вызывает POST /requests/ID/attachments)[185]. Для данных акта – либо создать Service Report автоматически: некоторые системы при закрытии заявки генерируют “заглушку” Service Report с кратким описанием[186]. Вероятно, в нашей системе план такой: не разрешать закрыть без отчёта, но дать возможность через бот создать упрощенный Service Report.
        ◦ /close <ID> – менеджер закрывает заявку. Бот вызывает PUT /requests/ID/status -> Closed. Доступно только если пользователь – PM или админ (иначе 403).
        ◦ /help – выводит список команд.
        ◦ /analytics – (для Admin) вывод ключевых метрик: открытые заявки, просроченные SLA, суммарная выручка за месяц, и т.д.[187][188]. Бот обращается к API (возможно, отдельный метод или внутренний) собирает данные из ERPNext (например, используя frappe.db.count и агрегаты, либо Prometheus) и отправляет сообщение директору. Это позволяет руководству получать сводки прямо в мессенджере.
    • Инфраструктура бота: Токен бота (BotFather) хранится в telegram_bot_token (site_config или settings)[9]. Сам бот – приложение, запущенное параллельно с ERPNext, которое может либо:
        ◦ работать внутри того же процесса (на Python, в фоне) – но это сложнее управлять;
        ◦ либо быть частью FastAPI сервиса. В System Overview указано, что бот реализован на Aiogram (Python) и “runs as part of custom backend service”[189]. Значит, FastAPI при старте поднимает asyncio-цикл бота. Он подключается к Telegram (polling или webhook) и слушает сообщения. При сообщении – определяет команду и вызывает соответствующий handler, который уже дергает либо прям функции Frappe (если этот бэкэнд имеет доступ к frappe ORM) либо делает HTTP запросы к API ERPNext. В нашей реализации, возможно, используется прямой доступ: т.к. бэкенд находится “рядом” с Frappe (тот же Python env), можно вызывать frappe.get_doc и т.д.
    • Безопасность бота: Требуется сопоставить учетные записи Telegram с пользователями системы. Для этого, вероятно, используется команда /register или /start, где бот выдает пользователю код, который надо внести в ERPNext профиле, либо просит ввести логин/пароль. Более простой путь: использовать отдельный токен авторизации. Например, выдать каждому инженеру и клиенту API Key/Secret или одноразовый код. Но в проекте не указано явно. Скорее всего, сделали вручную: админ сохраняет в настройках соответствие Telegram user ID -> ERPNext user. Тогда бот зная ID чата может понять, кто пишет. (В send_message у нас chat_id, а обратная ситуация – чтобы понять кто отправил, Aiogram message.from_user.id сопоставляется со списком).
    • Изоляция окружений: Можно завести двух ботов – один для теста, другой для прода, с разными токенами, и указывать их в настройках. Чтобы тестовые уведомления не уходили в боевые чаты.
    • JWT аутентификация (REST API): Для интеграции с ботом и внешним React-приложением реализована JWT-база аутентификации. Пользователь может получить JSON Web Token для обращения к API вместо использования cookie сессии. Основные характеристики (файл api/auth.py):
    • Получение токена: реализован метод login(username, password) (whitelisted, allow_guest)[190]. Он проводит обычную ERPNext-аутентификацию через LoginManager (то есть проверяет имя пользователя и пароль, а также 2FA если настроено)[191]. При успешном логине генерируется JWT: payload содержит sub: username, время выдачи (iat) и время истечения (exp) через 1 час[192]. Секрет для подписи берется из настроек (jwt_secret)[193]. Алгоритм HS256 (симметричная подпись). Если JWT отключен настройкой или библиотека не установлена – бросается ошибка[194].
    • Использование токена: после получения, клиент (бот или фронт) посылает запросы с Header Authorization: Bearer <token>. В hooks.py подключён before_request обработчик jwt_before_request[195]. Он на каждый запрос ERPNext проверяет: если заголовок Authorization с Bearer есть, пытается декодировать токен с секретом[196]. Если успешно – устанавливает пользователя с токена (frappe.set_user(user)), тем самым “логинит” запрос от имени этого пользователя[197]. Это работает для вызовов методов API (как frappe.whitelist методов или REST GET/POST на DocType), позволяя обходиться без полноценной web-сессии.
    • Время жизни и refresh: В спецификации указано “короткий TTL, refresh flow и scopes”[198][199]. Сейчас реализовано TTL ~1 час для токена[192]. Механизм refresh_token в коде явно не представлен, но планируется: обычно выдают вместе с access-токеном долгоживущий refresh-токен (сложноотгадываемый идентификатор), хранят его на сервере (в Redis или БД). Endpoint /auth/refresh может принять refresh-токен и выдать новый JWT[200]. Пока можно описать как часть ТЗ: добавить таблицу или Single API Token для refresh, либо использовать стандарт Frappe API Key/Secret.
    • Scopes и ограничение прав: JWT по сути несет только пользователя. Ограничение прав идет стандартно через роли пользователя (на сервере всё равно проверяются Role Permissions на вызываемые методы). При желании можно внедрить claim “roles” или “scope” в токен, и проверять его в jwt_before_request – напр., токен, выданный боту, может содержать scope=“bot” и тогда позволять только определенные методы. В описании сказано, что “бэкэнд проверяет роль пользователя на каждом запросе и может фильтровать данные по JWT”[201]. Значит, помимо стандартных ERPNext проверок, во внешнем FastAPI слое, перед выдачей ответа, еще раз проверяется: если токен от клиента, то возвратить только его данные (например, фильтр customer). Это фактически дублирование PQC на уровне API – но лучше перестраховаться.
    • Безопасность JWT: Секрет хранится в настройках (в site_config.json на проде), его знают только сервер и в ideal – он разный на dev/prod. Токен подписан, проверяется на каждый запрос. Использование HTTPS (TLS) шифрует токен в передаче. Также для особо важных пользователей 2FA требуется при получении (LoginManager уже учтет, либо можно требовать OTP code параметром). Rate limiting: Endpoint /auth/login должен быть ограничен по числу попыток (во FastAPI можно добавить), чтобы не было перебора пароля[202].
    • Пример использования: Телеграм-бот при старте, зная мэппинг chat->user, мог бы сразу получить JWT от имени этого пользователя (возможно используя сохраненный refresh-token или отдельный сервис-токен) – либо бот обходит JWT и вызывает frappe.call напрямую (если он внутри бench). React-frontend – при логине пользователя через форму, вызывает /api/method/ferum_custom.api.auth.login, получает token и хранит его (например, в localStorage), далее при запросах к своим API (на FastAPI) вкладывает этот JWT. Сервер на FastAPI либо проксирует к Frappe (с этим JWT) либо использует frappe объекты.
    • Мониторинг (Prometheus/Grafana): В систему встроены метрики для мониторинга производительности и состояния. Вероятно, это реализовано на стороне FastAPI: вставлен endpoint /metrics, который отдает статистику в формате Prometheus (text/plain). Используется библиотека prometheus_client в Python для подсчета, либо встроенный Frappe Monitor. Например, метрика ferum_requests_open_total – число открытых заявок (обновляется ежечасно скриптом, либо рассчитывается на лету запросом)[203]. Метрики запросов: FastAPI автоматически может собирать количество вызовов API, время обработки, ошибки – и отдавать их. Prometheus-агент (в контейнере) опрашивает этот endpoint каждые N секунд[203]. Настроенные графики (через Grafana) могут показывать: количество открытых заявок во времени, среднее время закрытия (вычисляется или отдельным gauge обновляется), загрузку CPU/Memory контейнеров, частоту ошибок (500 HTTP) и т.д.[203][204]. Кроме того, ERPNext можно мониторить стандартно (например, pings, scheduler health). Развертывание проекта включает docker-compose с сервисом Prometheus + Grafana, настроенным на наш backend.
    • Sentry (Отслеживание ошибок): Интегрирован клиент Sentry.io для централизованного сбора исключений и ошибок. В конфигурации указывается DSN sentry (site_config или .env), и при старте бэкенда Sentry SDK инициализируется[205]. В ERPNext-части можно подключить обработчик в hooks app_error_handler или просто использовать frappe.log_error, который потом можно экспортировать. Однако, скорее всего, основной сбор исключений идет с FastAPI (он ловит все unhandled exceptions и шлет в Sentry)[206][205]. Примеры: сбой вызова Google API, ошибка в бот-команде – сразу летит уведомление разработчикам через Sentry (с бэк-трейсом)[205][207]. Это ускоряет отладку: разработчики видят проблему практически в реальном времени. Также можно подключить фронтенд Sentry для React, чтобы ловить ошибки UI.
    • Email (SMTP): Хотя явно не перечислен, ERPNext по умолчанию использует email. В нашей системе, помимо Telegram, настроены и Email-уведомления: например, notify_project_manager отправляет email менеджеру при новой заявке[44], send_sla_breach_notifications отправляет email всем ответственным о просрочке SLA[116], notify_on_subcontractor_invoice – уведомляет email группу бухгалтеров[93]. SMTP-сервер настраивается в ERPNext (например, корпоративный Gmail). Шаблоны писем можно оформить через стандартный Notification DocType (в фикстурах они включены – все Notification относящиеся к Ferum Custom)[208]. E-mail служит дублирующим каналом, более официальным (на него полагаются для архивов).
    • WhatsApp (потенциально): В документации упомянуто, что, возможно, будет интеграция и с WhatsApp, так как не все клиенты пользуются Telegram[209]. WhatsApp не позволяет ботам напрямую, поэтому вариант – через WhatsApp Business API (Twilio или Infobip). На этапе ТЗ можно заложить: отправка основных уведомлений клиентам – через WhatsApp Cloud API (шаблоны сообщений). Например, клиенту можно автоматом отправить WhatsApp: “Ваша заявка №123 выполнена, спасибо за обращение”. Пока это не реализовано, но архитектура может предусмотреть модуль отправки WhatsApp (подобно telegram.send_message, но через HTTP API WhatsApp). Это требует дополнительных токенов и ID шаблонов в настройках.
    • Интеграция с 1С (будущее): В перспективе, для синхронизации клиентов и счетов с бухгалтерией 1С (или KUB-24) может потребоваться экспорт данных в формате, понятном 1С – XML, CSV или через REST-сервис 1С[210]. На сейчас это не сделано, но ТЗ может указать: система готова к интеграции – все данные о счетах, актах, клиентах хранятся централизованно, их можно выгрузить отчётами или напрямую SQL. Возможно, будут реализованы скрипты экспорта (например, ежемесячно генерировать CSV всех оплаченных счетов клиентов для импорта в 1С).
API (REST интерфейсы)
Для сторонних приложений и ботов разработан слой API. Часть реализована непосредственно в Frappe (whitelisted методы), часть – во внешнем FastAPI. Основные endpoints и их поведение определены в документе API Design:
    • Список и детали объектов:
    • GET /projects – получить проекты. Поддерживает фильтрацию по клиенту или статусу. Реализовано ограничение: возвращает только проекты, доступные этому JWT-пользователю (например, PM – только свои, Client – только свои, Admin – все), это достигается либо наложением фильтров на основе roles, либо через PQC на уровне прямого frappe.get_list. Вероятно, во внешнем API вызывается frappe.get_list("Service Project", filters=..., fields=[...]), а перед этим дополняются filters на company/customer, исходя из JWT claims[211][212]. GET /projects/{id} – детали одного проекта (с перечислением объектов, счетов, краткой статистикой) – требует, чтобы вызывающий имел доступ к этому проекту (проверяется тоже либо через has_permission, либо кодом).
    • POST /projects – создать новый проект. Только PM или админ. Бэкенд либо дергает frappe.get_doc({...}).insert()[213], что триггерит все серверные валидации (например, уникальность объектов), либо использует Frappe REST. После создания возвращает ID.
    • PUT /projects/{id} – обновить проект (например, продлить дату, сменить статус). Проверяет права (PM своего проекта или админ). Может быть запрещено менять проект_manager. Если нужно добавить объект – можно отдельным endpoint или послав вместе изменённый child table.
    • Service Objects API: (используется редко внешне)
    • GET /objects?customer=XYZ – получить объекты клиента.
    • POST /objects – клиент теоретически может добавить новое оборудование (если, например, установил новую систему и хочет включить её). Но часто это делает Office Manager. Можно предусмотреть: если вызов от клиента, автоматически проставить customer из его профиля и создать объект.
    • Service Requests API:
    • GET /requests?status=Open – получить список заявок. По умолчанию возвращает в соответствии с ролями: для инженера – либо все ему назначенные (можно по умолчанию подставить filter assigned_to=me), для клиента – только его (filter customer или owner), для PM – только по его проектам. Скорее, сделано проще: list_service_requests метод в ferum_custom.api.service – он берет фильтр status, и если текущий пользователь Website User или роль Client – добавляет filters["owner"] = user[56]. Но лучше в REST бэкенде сделать универсальнее: смотреть roles из JWT и накладывать нужное (как в PQC). Ответ – JSON со списком (поля ID, title, status, priority, etc, отформатировано для мобильного UI)[214].
    • GET /requests/{id} – детали конкретной заявки (включая описание, статус, объект, связанный отчет и вложения). Проверяется доступ: должен пройти has_permission (иначе 403). Реализовано как get_service_request(name) в API – возвращает doc.as_dict()[215], т.е. все поля. В портале используется именно этот метод[216].
    • POST /requests – создать новую заявку. Используется ботом и порталом. В портале метод create_service_request(title, description, service_object) – создает документ через frappe.new_doc и insert[217]. Аналогично, REST API может принимать JSON, в котором достаточно title, description и id объекта (или поля клиента). Если JWT-пользователь – клиент, сервер может автоматически подставить его Customer (если система знает связку User->Customer), чтобы заявка привязалась. Если объект указан – Customer/Project проставятся на сервере при вставке (через before_insert). Этот вызов триггерит и почтовое, и телеграм-уведомление (notify_project_manager) – то есть, интеграции работают, как если бы заявка создана в UI.
    • PUT /requests/{id} – обновить заявку (например, отредактировать описание или назначить инженера). Обычно для внешних пользователей недоступно, могут использовать внутренние (Office Manager – поправить опечатку, PM – назначить кого-то). Ботами это может не использоваться, т.к. проще отдельные действия.
    • PUT /requests/{id}/status – сменить статус заявки[182]. Получает, например, {"status": "Completed"}. Сервер на стороне FastAPI:
        ◦ Загружает документ или через Frappe API вызывает frappe.get_doc("Service Request", id), проверяет текущий статус.
        ◦ Выполняет логику, аналогичную validate_workflow_transitions: если запрос от инженера и пытается Closed – вернуть 403 (не разрешено); если Completed без отчёта – вернуть 400 с сообщением “Service report required”[184];
        ◦ Иначе применяет: doc.status = NewStatus; doc.save() (или apply_workflow/submit). Система сама вызовет on_update – проставит время и т.д.
        ◦ При успехе – возвращает 200. В бот-диалоге предусмотрены эти проверки: как описано, если получить ошибку про отчёт, бот может попросить заполнить отчёт.
    • Дополнительно: POST /requests/{id}/attachments – загрузить файл к заявке. Ожидает файл (multipart) и параметр description. API создает CustomAttachment, привязывает к заявке (RequestPhotoAttachmentItem) и инициирует upload to Drive. Бот дергает это при команде /upload_photo. В проекте прямо не написано, но добавить несложно: frappe.get_doc({"doctype": "CustomAttachment", file_name:..., file_url:...}).insert() и связать.
    • Service Reports API:
    • GET /reports?project=X – список актов, фильтр по проекту или статусу. Наверное, доступно админам и PM. Инженеру не особо нужно, клиенту – возможно нет (кроме как скачать свой).
    • GET /reports/{id} – получить подробности акта (полей, включая work_items и документы). Может использоваться для, например, отобразить клиенту акт (клиенту можно дать доступ только если doc.customer == его customer, либо проще – клиент смотрит PDF).
    • POST /reports – создать акт. Это сложный объект (с таблицей), поэтому через бот, возможно, не делается. Но, например, если сделать мобильное приложение для инженеров, они могут полностью заполнить акт в поле. Тогда API ожидает JSON: {service_request: ID, work_items: [...], documents: [...]}. Бэкенд создает doc = frappe.new_doc("Service Report", ...) + doc.insert(). Либо можно упростить: Draft Report может создаваться автоматически, когда инженер жмет “Done” без акта – как вариант, backend создаст Service Report с минимумом (из заявки возьмет описание проблемы как описание выполненных работ, hours=..., rate=..., total=0), чтобы разблокировать закрытие. Но это тонкости процесса, зависящие от бизнес-решения.
    • PUT /reports/{id} – обновить/добавить вложения. Например, после подписания акт сканируется – можно вызвать /reports/id с attachment, чтобы прикрепить. Или сменить статус (если есть свой Workflow, но обычно акты утверждаются внутри ERPNext, не через внешний API).
    • Invoices API:
    • GET /invoices?project=X – список счетов (для PM – только своего проекта, для бухгалтера – всех).
    • GET /invoices/{id} – детали счета (поля, ссылки на сканы). Клиентам этот API не доступен.
    • POST /invoices – создать счёт. Используется, например, офис-менеджером через React-интерфейс: вместо формы ERPNext, человек заполняет веб-форму, и вызывается API. JSON включает project, counterparty_type, counterparty_name, amount, date, etc. На сервере: создает новый Invoice doc, insert. Это вызовет after_insert: синхронизация с Google Sheets, уведомления, так что и Admin узнает, что счёт добавлен[89]. Важно проверять: если роль вызывающего – PM, то он может только свой проект; если Office Manager – любой проект, но у него должна быть роль (стоит убедиться, что JWT содержит Office Manager).
    • PUT /invoices/{id} – обновить (например, прикрепить номер счета из бухгалтерии, или изменить статус). В основном, выделяют:
        ◦ PUT /invoices/{id}/status – смена статуса (Send/Paid). Например, бухгалтер приложением выбирает счет и нажимает “Mark as Paid”, фронт зовет этот endpoint с {"status": "Paid"}. Сервер проверяет: пользователь – Chief Accountant или Admin, иначе 403[218]. Если ОК – ставит doc.status = Paid, doc.docstatus = 1 (т.е. проведен), вызывает doc.save/submit. Это триггерит on_update -> sync_to_google_sheets (отметит оплату)[106], также update_project_financials (пересчитает прибыль проекта)[105].
    • Примечание по безопасности: Все API защищены JWT, и на уровне кода проверяется роль/права. Например, bulk_mark_sent (вызывается в UI, но можно было и через API) – прямо проверяет roles на пересечение с разрешенными[86]. Подобную проверку делает и FastAPI: или декораторами (e.g. @roles_allowed("Project Manager", "Office Manager")). JWT payload может содержать roles, либо backend сам может вызывать frappe.get_roles(user) из БД.
    • Прочие API:
    • POST /auth/login – уже описано, получает JWT[219]. В payload передаётся `otp` (one-time password), если для пользователя включён 2FA – без него запрос вернёт ошибку. Для пользователей без 2FA поле можно опустить.
    • POST /auth/refresh – обновление JWT (ожидается refresh_token)[200]. Нужно спроектировать: выдавать refresh_token при логине (можно его генерировать и возвращать вместе с токеном, как return {"token": ..., "refresh": "<GUID>"}), хранить refresh где-то (в Ferum Custom Settings или отдельном DocType), и при /refresh проверять и выдавать новый JWT.
    • GET /users/me – вернуть профиль текущего пользователя (имя, роли, возможно привязанный Customer или Employee). Нужно для фронтенда, чтобы знать, какие разделы ему показывать.
Таким образом, API охватывает все ключевые сущности, что позволяет построить внешние интерфейсы (чат-бот, мобильное приложение, интеграции IoT). При этом, каждый вызов API проходит через те же бизнес-правила ERPNext, что и пользователи Desk, благодаря тому, что backend использует либо Frappe ORM напрямую, либо REST Frappe – тем самым триггеры (валидации, on_update) срабатывают и интеграции (Sheets, Drive) выполняются автоматически[221][222].
Пользовательский интерфейс (Portal, Desk, мобильный доступ)
    • ERPNext Desk (для внутреннего использования): В модуле Ferum Custom настроены пользовательские формы и списки с улучшенным UX:
    • Service Project: добавлены кнопки действий на форме – “Activate Project”, “Complete Project”, “Cancel Project”, “Reopen Project” (см. service_project.js) для смены статуса через Workflow[16]. Эти действия появляются в меню Change Status в зависимости от текущего статуса (условия в коде)[223]. Аналогичные действия продублированы на ListView, чтобы менеджер мог массово управлять проектами (код service_project_list.js)[224].
    • Также на форме проекта добавлены кнопки Create -> Customer Invoice и Create -> Subcontractor Invoice – они открывают быстрое создание нового документа Invoice с предзаполненным проектом и типом контрагента[225][226]. Это ускоряет работу PM/OfficeManager: не надо вручную выбирать проект и тип, система сама подставит. Эти же функции доступны и из спискового вида проектов (Actions)[227][228].
    • Service Request форма может иметь клиентские скрипты для удобства: например, при выборе Service Object сразу вызывается бекенд get_engineers_for_service_object и в поле assigned_to предлагает инженера. Сейчас явного service_request.js нет, но service_object автозаполняет customer и project в серверном методе, что отображается после сохранения. Можно добавить refresh-скрипт: если заявка без отчёта и статус Completed, показывать предупреждение (но это и так запрещено).
    • Service Report форма – здесь важно контроль вложений: validate_attachments запрещает submit без прикрепленных файлов в таблице Documents[66]. Возможно, на форме Service Report сделано поле “Attach PDF” – но у нас PDF генерится после сабмита. Можно, однако, добавить кнопку “Print PDF” для предпросмотра акта перед сабмитом. В репозитории упомянут Print Format (фикстуры содержат Print Format для Ferum Custom)[229], т.е. кастомный шаблон акта. Эта печатная форма используется get_print. Она должна быть настроена под фирменный бланк (в фикстуре Print Format likely “Service Report”).
    • Invoice форма – также упомянут invoice.js и invoice_list.js. На форме, вероятно, скрипт динамически меняет лейблы/видимость полей: напр., если counterparty_type = Customer, показать поле “Customer (client)” и скрыть поле “Supplier”, и наоборот. Или отметить, что при выборе Paid статус – требовать docstatus=1. В invoice_list.js можно добавить групповые действия: “Mark as Sent” – для выбранных (bulk_mark_sent), “Mark as Paid” – но в нашем коде bulk для Paid не реализован (потому что оплату по одному лучше). Тем не менее, bulk_mark_sent(names) реализована[230] и, вероятно, вызывается из списка Invoice (например, массово отметить 10 счетов как отправленные). Роли проверяются внутри, но UI тоже можно отфильтровать, чтобы кнопка была видна только тем, кому положено (в frappe.listview_settings можно прописать условие roles).
    • Workspace и отчетность: Фикстуры содержат Workspace (рабочий стол) модуля Ferum Custom[231], значит, в ERPNext есть отдельная страница с ярлыками на DocType (Projects, Requests, Reports, Invoices, etc.) для удобства. Также Dashboard Chart – настроены диаграммы: “Открытые заявки по статусам” (пирог) и “Счета по проектам” (столбцы)[232][233]. Они отображаются либо на Workspace, либо на Dashboard. Например, для менеджера можно вывести график заявок, разбитых по статусам прямо на рабочем столе, чтобы он видел, сколько сейчас Open/In Progress.
    • Уведомления в Desk: Кроме Telegram/Email, в ERPNext можно задействовать систему уведомлений: Notification doc “SLA Breach” – чтобы в шторке уведомлений ERPNext появлялись сообщения при нарушении SLA. В фикстурах Notifications, вероятно, есть такие настройки[208]. Также, todo: возможно, при назначении инженеру – ему создаётся ToDo (Frappe имеет Assignment feature, но здесь просто поле). Можно подумать: если engineer – Employee, то можно создать Assignment запись. Но пока не указано, вероятно, достаточно Telegram уведомления.
    • Веб-портал для клиентов: Для роли Client настроены простые страницы портала:
    • /portal – главная страница клиента (вероятно, список его заявок). Repo содержит portal/service-request.html и portal/new-request.html. Судя по коду, portal страница выводит список (возможно с помощью генерации стандартной портальной страницы). Неявно, но list_service_requests возвращает data. Возможно, Ferum Custom зарегистрирован как веб-портал (Portal Settings, Module Def в фикстурах)[234]. Если клиент заходит, он видит список своих заявок (можно отфильтровать). В нашем коде явно страницы:
        ◦ Новая заявка: /portal/new-request – HTML-шаблон с формой: поля Title, Description, optional Service Object (строкой)[235][236]. При нажатии Submit вызывается frappe.call("ferum_custom.api.service.create_service_request", {title, description, service_object})[179]. Если вызов успешен, выводится сообщение с ссылкой на подробную страницу заявки[237]. Таким образом, клиент может через веб создать тикет без доступа к Desk.
        ◦ Просмотр заявки: /portal/service-request?name=XXX – шаблон service-request.html отображает детали: заголовок с номером и названием, статус и описание[135]. Если у заявки есть linked_report, то отображается блок с ссылкой “Download Service Report” – сформированной так, чтобы скачать PDF печатной формы через Frappe API (/api/method/frappe.utils.print_format.download_pdf?doctype=Service Report&name=...)[136]. Это позволяет клиенту получить акт по своей заявке прямо из портала. Кнопка появляется только если акт привязан[136]. Внизу – ссылка назад к списку.
        ◦ Список заявок: Явно HTML нет, но, возможно, используется стандартный Template List из ERPNext (Portal Settings). Либо страница portal/index настроена. Если нет, можно реализовать: в Workspace портала для клиента – вывести список его Service Request. Упрощенно: можно на /portal перенаправить на стандартный /requests page, но т.к. Service Request – нестандартный (не Issue), написали свои. Возможно, portal.js есть, но не критично.
    • Аутентификация клиента: Клиенты заводятся как Website Users, им выдаются логин/пароль. Либо админ сам создает User, даёт роль Client и связку с Customer (через Contact или custom link). Затем клиент через /login ERPNext заходит. Можно было реализовать OAuth с Google, но не указано, значит, обычная регистрация.
    • UI/UX портала: Все лишнее скрыто – у клиента нет доступа к Desk, только portal menu, где показываются, например, “My Service Requests”, “New Request”. Этот портал – на базе стандартного frappe web templates, стилизовать можно при необходимости, но, видимо, хватит стандартного Bootstrap.
    • Мобильный доступ: Основной мобильный интерфейс – через Telegram/WhatsApp бота (описан выше). Кроме того, Frappe имеет responsive UI, но полная Desk для инженера избыточна. Поэтому предполагается, что инженеры не работают через Desk на телефоне, а используют бот (команды, отправка фото). Тем не менее, можно открыть портал и с телефона – для клиентов это важно (например, они могут отправить заявку с мобильного браузера).
    • Отчётность и дашборды: Для руководства планируются отдельные отчёты: например, отчёт “Просроченные заявки”, “Незакрытые заявки > 5 дней”, “Прибыль по проектам” – их можно реализовать либо через стандартный Report (SQL query or Script Report), либо полагаться на Google Sheets и Grafana. В фикстурах Report указано, что сохранены отчёты модуля[238], вероятно: Open Service Requests, Project Profitability и т.п. Руководитель (Director) может просматривать их в ERPNext или запрашивать через бот, как упомянуто.
SLA, уведомления и аналитика
(Данный раздел акцентирует, как система обеспечивает соблюдение SLA и предоставляет данные для управления качеством сервиса):
    • Поддержка SLA: В каждом Service Level Agreement (неявно заданном правилами приоритета) система вычисляет крайние сроки реагирования[40]. Эти правила можно вынести в настройки (например, High-Emergency = 4h, Medium-Emergency = 8h, High-Routine = 24h, Normal = 72h). Сейчас они зашиты, но можно сделать поля в Ferum Custom Settings для гибкости. Срабатывание контроля реализовано двумя способами: 1) Онлайн: при сохранении заявки, если обнаружено, что уже просрочена – check_sla_breach мгновенно сообщает (на случай, если кто-то долго не менял статус)[239]; 2) Фоново: ежечасный cron вызывает check_all_slas – проходит по всем открытым заявкам и вызывает check_sla_breach[112]. Последний подход страхует, если заявки не редактировались (чтобы не пропустить просрочку).
    • Эскалации: Если SLA нарушен, происходят уведомления (описано в разделе интеграции): email соответствующим ролям (PM, Office Managers)[115], Telegram-оповещение в общий чат[117]. Можно расширить: повторное уведомление каждые N часов пока заявка не взята в работу (но это можно реализовать, например, меткой “Escalated” и ещё одним scheduler). Пока однократное уведомление. Менеджеры при получении, как правило, вручную связываются с инженером или перераспределяют заявку.
    • Нормативы и запреты: SLA реализуется не только по времени, но и логикой запретов: не позволено закрыть заявку без отчёта[34] – это гарантия качества, что по каждой выполненной работе есть акт. Не разрешено ставить “In Progress” без назначения – гарантия, что кто-то ответственный определён. Закрывать заявку может только менеджер – чтобы клиент/инженер не закрыл самовольно, пока менеджер не проверит качество. Эти правила – часть SLA-политики (контроль статусов).
    • Аналитика SLA: Система рассчитывает duration_hours выполнения, а также можно добавить вычисление response_time – время от открытия до начала работы (когда статус перешёл в In Progress, у нас есть actual_start_datetime). Эти метрики могут храниться в полях заявки или вычисляться отчётами. На их основе строятся KPI: % заявок выполненных до SLA, среднее время реакции, среднее время решения. Пример: Telegram-команда /analytics может вернуть: “Всего открытых заявок: 5, из них 2 просрочены SLA. Среднее время решения: 16 часов. В этом месяце выполнено заявок: 30, соблюдение SLA: 90%.” – такие цифры бэкенд может вычислить, делая агрегаты по данным.
    • Фотофиксация и контроль качества: Каждый Service Request сопровождается фотографиями (до/после) в Attachments[240]. Это служит не только для клиента (доверие, прозрачность), но и для внутреннего контроля качества. Руководитель может выборочно просматривать отчёты и фото к ним, чтобы убедиться в качестве работ. С помощью Google Drive эти фото доступны менеджеру (если нужно крупно рассмотреть). Сами фото (CustomAttachment) также можно использовать в отчетах (например, отчет “Список работ с фото До/После” – для презентации клиенту).
    • Логирование и аудит: Все важные события (создание, изменение статуса, входы в систему, ошибки) логируются. Для SLA-анализа можно вести SLA log – но даже без него у нас есть timestamp статусов через Version и поля actual_start/end. Администратор может осуществлять аудит: в Error Log будут записи “SLA Breach Alert for SR-0001”[114], “Telegram send failed” или stacktrace интеграций[159]. Эти логи, как описано, могут ежемесячно архивироваться на Google Drive для долговременного хранения[241]. То есть, скрипт (cron job) может собирать логи и сохранять csv в защищённую папку. Это повышает наблюдаемость: даже спустя время можно расследовать, были ли сбои уведомлений или нарушения процесса.
    • Dashboard SLA: Через Grafana или ERPNext Dashboard можно визуализировать SLA: метрика “Active SLA breaches” (количество сейчас просроченных), график “SLA Breaches per month”. Это поможет менеджменту отслеживать тренды и принимать меры (например, если часто High Priority заявки просрочены – надо усилить дежурство).
Логирование, мониторинг и безопасность
    • Структурированные логи: В модуле используется frappe.logger() и frappe.log_error() для фиксирования ошибок с контекстом[159][242]. Например, при сбое Drive API или Telegram API логируется traceback с понятным заголовком (“Drive upload failed”, “Telegram send failed”)[243][242]. Это позволяет админам при просмотре логов сразу видеть категорию проблемы. Также, метод frappe.logger().exception(...) записывает полную traceback, что полезно для отладки. Предусмотрено логирование предупреждений: например, если Telegram не настроен, выводит warning “Message suppressed”[176], если SLA breach – warning, если счёт без получателей уведомления – warning[94]. Все эти логи хранятся в таблице Error Log или в файлах журналов, которые можно агрегировать. Можно настроить JSON-фомат логов – Frappe по умолчанию пишет текстом, но если запускать под supervisord, можно перенастроить format на JSON (чтобы интегрировать с ELK, например). Пока это не явно сделано, но в ТЗ можно указать: “Использовать структурированный формат логов (JSON) для интеграции с централизованным сбором логов.”
    • Мониторинг производительности: Как указано, включен Prometheus endpoint[203]. Там можно собирать: количество запросов API, время их обработки, usage memory, и бизнес-метрики (количество заявок). Grafana-дашборды помогут операторам видеть здоровье системы: например, график времени ответа на API – если растет, значит, надо проверить сервер.
    • Sentry (ошибки): Все непойманные исключения отправляются на Sentry[244]. Разработчики настроили алертинг, т.е. при появлении новой ошибки – приходит уведомление. В Sentry остаются стектрейсы, локальные переменные – это ускоряет исправление багов. Например, если sync_to_google_sheets падает из-за конкретного формата данных, разработчик увидит ошибку и может быстро выпустить патч.
    • Audit Trail: ERPNext везде включает версионирование (флаг track_changes), а также хранит историю входов пользователей. По требованиям безопасности, критичные действия – такие как пометка счета оплаченным, удаление документа – фиксируются. Можно дополнить: подключить frappe.audit logger, чтобы записывать, например, в отдельную таблицу все изменения финансовых полей (кто и когда изменил статус счета). Но даже без этого:
    • Все документы имеют Owner (кто создал) и Modified By.
    • Time Log (Comments) указывает, кто сменил статус (через add_comment)[49].
    • Можно настроить ERPNext Activity Log или Notification на удаление документов (System Manager получит письмо, если кто-то удалил заявку).
    • Безопасность данных: В системе внедрены стандартные меры: хеширование паролей (ERPNext), 2FA для админов[147], ограничение доступа по ролям (описано выше). Дополнительно, возможна шифрация чувствительных полей: например, если хранятся паспортные данные сотрудников или токены – можно пометить поле Protected (ERPNext будет шифровать в БД). Например, jwt_secret можно хранить как Encrypted поле Single DocType (тогда он шифруется). Резервные копии базы (SQL бэкапы) – делаются ежедневно и выгружаются на Google Drive (в отдельную папку, защищенную, с ограниченным доступом)[245]. В ТЗ указано: резервные копии шифруются перед загрузкой или хранятся в безопасном месте (сервисный аккаунт Google Drive имеет доступ, и только админ знает его ключ)[245][246]. Таким образом, даже компрометация Google Drive через интерфейс не даст расшифровать данные.
    • Сетевые настройки: Приложение развёрнуто за обратным прокси (NGINX) с SSL – все соединения только по HTTPS[247]. Используются современные параметры TLS, чтобы данные (включая JWT токены, пароли) не шли в открытом виде. Nginx также ограничивает доступ к бэкенду (можно закрыть /api/method/frappe.* для внешнего мира, оставив только нужные кастомные API).
    • Rate Limiting: Для API авторизации и, возможно, для всех API, включена защита от брутфорса и DoS. Например, на /auth/login – не более 5 запросов в минуту с одного IP[202]. Это можно реализовать через SlowAPI библиотеку (для FastAPI) или через Nginx limit_req. Аналогично, бот команд может быть злоупотреблен – но Telegram сам ограничивает ботов.
    • Разделение окружений: Тестовая среда и боевая – изолированы. Разработку ведут в отдельной копии, с тестовыми данными. Там могут использоваться тестовые ключи интеграций (например, другой Telegram бот). Перед развертыванием на прод, проходят код-ревью (репозиторий под контролем), CI/CD (GitHub Actions) проверяет сборку, запускает unit-тесты (если написаны). В релизе, Sentry DSN может отличаться, чтобы ошибки dev не смешивались.
    • Дашборды мониторинга: Администратору предоставляется панель Grafana с основными графиками: нагрузка CPU/RAM, кол-во запросов, ошибок, и бизнес-графики (открытые заявки). Также настроены оповещения: если, к примеру, ошибок > X в час – Sentry/Grafana могут прислать алерт админу.
    • Обновление/патчи: В репозитории есть patches.txt – список миграций, которые применяются при обновлении версии приложения (например, добавление новых полей). Это гарантирует, что при внедрении новых функций (JWT refresh, новые DocTypes) база обновится без сбоев.
Наконец, вся данная техническая спецификация будет передана команде разработки, чтобы они реализовали (и доработали существующую) функциональность модуля ferum_custom с учётом описанных требований, следуя принятому стеку Frappe/ERPNext и лучшим практикам по безопасности, наблюдаемости и качеству кода. Каждый компонент – от архитектуры многокомпанейности до интеграции с Google и Telegram – спроектирован так, чтобы система была надежной, масштабируемой и удобной для всех ролей пользователей. Приведенные детали (имена файлов, функций, поля DocType) позволят разработчикам быстро находить нужные места в коде и вносить изменения согласно требованиям.
